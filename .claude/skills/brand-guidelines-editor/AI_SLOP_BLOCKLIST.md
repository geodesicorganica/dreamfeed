# AI Slop Blocklist

Use this as a rejection checklist when editing Stakeport brand guidelines, Dreamfeed UX rules, website design standards, or product UI guidance.

## Research Rule

Before adding or revising brand/design rules, check whether the proposed pattern resembles low-effort AI-generated output. Use repo evidence first, then current design-practice sources, then founder interrogation for unresolved judgment calls.

## Blocked Visual Patterns

- Decorative gradient orbs, bokeh blobs, glow clouds, or abstract background smears with no product meaning.
- Thin colored stripes on cards used as the main hierarchy instead of real structure, state, or grouping.
- Repeated rounded cards with identical spacing, identical metadata, and no operational prioritization.
- Fake dashboards: charts, graphs, counters, badges, or sparklines that do not map to real data or source provenance.
- Overly smooth "premium SaaS" surfaces that hide status, timestamps, owners, blockers, or source files.
- Decorative icons used where a status, action, file, command, or data point should be shown.
- One-note palettes dominated by a single hue family without functional color semantics.
- Generic AI-generated hero visuals, abstract mesh gradients, floating glass panels, crypto coin imagery, and people-smiling-at-screens.
- Dense UI that is merely crowded, not structured by priority, workflow, state, and scan path.

## Blocked Copy Patterns

- Generic SaaS claims: purpose-built, seamless, unlock, empower, comprehensive, next-generation, best-in-class, enterprise-grade.
- Vague AI phrases: AI-powered, intelligent insights, smart automation, copilot for everything, transform your workflow.
- Polished but empty summary text that sounds plausible while omitting mechanism, owner, source, date, constraint, or next action.
- Overconfident claims without provenance, especially performance, prediction, risk, market, or adoption claims.
- Repeated sentence frames that make every section sound interchangeable.
- Excessive enthusiasm, exclamation marks, emojis, crypto-native slang, or motivational filler.

## Blocked UX Patterns

- No timestamp for status surfaces that imply freshness, health, audit, validation, or "live" state.
- Status chips that all look equally important or use the same color treatment for different states.
- Approval items that cannot open the underlying file, surface, command, or review artifact.
- "Graph" surfaces that show lists of entities without an actual node/edge representation when graph inspection is promised.
- Cards that cannot be clicked even though they represent an object with source detail.
- Missing empty, loading, error, stale, blocked, complete, declined, and resolved states.
- Layouts that put general status above urgent approvals, blockers, or founder decisions.
- Navigation that mirrors implementation files rather than the founder's daily operating workflow.

## Required Countermove

When a blocked pattern appears, replace it with:

- A source-backed data element.
- A real workflow affordance.
- A clearer status model.
- A concrete timestamp or provenance marker.
- A simpler layout with fewer decorative elements.
- A product-specific rule that explains when the pattern is allowed and when it is forbidden.
